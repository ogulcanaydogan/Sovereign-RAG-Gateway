# v0.7.0-alpha.1 Release Notes

`v0.7.0-alpha.1` focuses on release-evidence integrity hardening. The API surface remains stable while release verification moves from "asset presence" to "cryptographic verification."

## Highlights

- `release-verify` now validates both:
  - `bundle.sha256` integrity for `bundle.json`
  - detached signature verification via `bundle.sig` + `release-evidence-public.pem`
- Release evidence generation now exports `release-evidence-public.pem` as a first-class release artifact.
- Added tamper tests for signature validation failure cases to prevent silent regressions.

## Included Components

- Release verifier:
  - `.github/workflows/release-verify.yml`
  - `scripts/check_release_assets.py`
- Evidence artifact generation:
  - `scripts/generate_release_evidence_artifacts.py`
- Tests:
  - `tests/unit/test_check_release_assets.py`
  - `tests/unit/test_generate_release_evidence_artifacts.py`

## Migration Notes from v0.6.x

### 1) Strict Release Verification

`release-verify` now runs with strict options:

```bash
python scripts/check_release_assets.py \
  --repo "<owner>/<repo>" \
  --latest \
  --verify-bundle-integrity \
  --verify-signature \
  --require-public-key
```

### 2) Expected Release Evidence Assets

Releases should now include:

- `bundle.json`
- `bundle.sha256`
- `bundle.sig`
- `release-evidence-public.pem`
- `release-evidence-metadata.json`
- `events.jsonl`
- `sbom.spdx.json`

## Compatibility

- No breaking changes in OpenAI-compatible API endpoints.
- Audit and policy contracts remain backward-compatible.
- Legacy newline digest compatibility is retained for older release artifacts during verification.

## Known Limitations

- Signature verification currently assumes OpenSSL-compatible RSA public key output.
- Weekly scheduled verification checks the latest release only; historical tag sweeps require explicit `--tag` runs.
